## 1. 项目概述

### 1.1 项目背景

内部审计是企业风险管理和内部控制的重要环节。传统审计工作依赖人工记录访谈、手动整理风险点、编写审查报告，效率低且容易遗漏关键信息。本系统旨在通过信息化手段和 AI 技术辅助审计人员，提升审计工作效率和分析质量。

### 1.2 项目目标

| 目标 | 描述 |
|------|------|
| 流程标准化 | 将审计项目、访谈记录、风险点、待办事项纳入统一管理 |
| 信息结构化 | 结构化存储访谈内容和风险信息，便于检索和分析 |
| AI 辅助分析 | 利用大语言模型自动提取关键发现、识别风险、生成审查报告 |
| 本地化部署 | 数据存储在本地，确保审计数据的安全性和保密性 |

### 1.3 目标用户

| 用户角色 | 说明 |
|---------|------|
| 审计管理员 | 管理审计项目、分配任务、审核报告 |
| 审计人员 | 执行访谈、记录发现、管理风险点 |
| 项目负责人 | 查看项目进度、审批风险缓释方案 |
| 系统管理员 | 管理用户账户、系统配置、数据备份 |

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 审计项目 | 一次完整的审计工作单元，包含项目信息、访谈记录、风险点等 |
| 访谈记录 | 审计人员与被审计方沟通的详细记录 |
| 被访谈人 | 接受审计访谈的人员 |
| 风险点 | 审计过程中识别的潜在风险 |
| 风险矩阵 | 5×5 的可能性-影响程度矩阵，用于风险可视化 |
| AI 分析 | 利用大语言模型对审计内容进行智能分析 |

## 2. 系统架构

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│                    Electron 桌面应用                   │
│                                                      │
│  ┌──────────────────────┐  ┌──────────────────────┐  │
│  │   渲染进程 (Vue 3)    │  │   主进程 (Node.js)    │  │
│  │                      │  │                      │  │
│  │  ┌────────────────┐  │  │  ┌────────────────┐  │  │
│  │  │  Element Plus  │  │  │  │  IPC Handler   │  │  │
│  │  │  UI 组件库      │  │  │  │  (main.mjs)    │  │  │
│  │  └───────┬────────┘  │  │  └───────┬────────┘  │  │
│  │  ┌───────┴────────┐  │  │  ┌───────┴────────┐  │  │
│  │  │  Vue Router    │  │  │  │  Database      │  │  │
│  │  │  Pinia Store   │  │  │  │  (SQLite)      │  │  │
│  │  └───────┬────────┘  │  │  └───────┬────────┘  │  │
│  │  ┌───────┴────────┐  │  │  ┌───────┴────────┐  │  │
│  │  │  Pages/Views   │  │  │  │  AI Service    │  │  │
│  │  │  Components    │  │  │  │  (LLM API)     │  │  │
│  │  └────────────────┘  │  │  └────────────────┘  │  │
│  └──────────┬───────────┘  └──────────┬───────────┘  │
│             │     contextBridge      │               │
│             └────────────────────────┘               │
└─────────────────────────────────────────────────────┘
                      │
                      │ HTTPS
                      ▼
        ┌─────────────────────────┐
        │    AI 提供商 API         │
        │  DeepSeek / Qwen / ...  │
        └─────────────────────────┘
```

### 2.2 技术架构分层

| 层次 | 技术选型 | 职责 |
|------|---------|------|
| 表现层 | Vue 3 + Element Plus | 用户界面、交互逻辑 |
| 状态层 | Pinia + Vue Router | 应用状态管理、页面路由 |
| 通信层 | Electron IPC + contextBridge | 前后端进程通信 |
| 业务层 | main.mjs (IPC Handlers) | 业务逻辑处理 |
| 数据层 | better-sqlite3 | 本地数据持久化 |
| 外部集成 | axios → LLM API | AI 能力接入 |

### 2.3 数据流

```
用户操作 → Vue 组件 → Pinia Store → window.api.xxx() → IPC → main.mjs
                                                              │
                                                    ┌─────────┴─────────┐
                                                    ▼                   ▼
                                              database.mjs        ai-service.mjs
                                              (SQLite CRUD)       (LLM API 调用)
                                                    │                   │
                                                    ▼                   ▼
                                              本地数据库文件       AI 提供商服务器
```

## 3. 功能需求

### 3.1 用户认证与权限管理

#### FR-AUTH-001 用户登录

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |
| 描述 | 用户通过用户名和密码登录系统 |

**功能规则：**

1. 系统首次启动自动创建默认管理员账户（用户名 `admin`，密码 `admin123`）
2. 密码使用 bcrypt 算法加密存储
3. 登录成功后将用户信息存入 `localStorage`，并生成认证 Token
4. 路由守卫自动检测登录状态，未登录用户重定向到登录页
5. 登录成功后记录最后登录时间

**输入/输出：**

| 输入 | 输出 |
|------|------|
| 用户名（字符串） | 登录成功：用户信息对象 |
| 密码（字符串） | 登录失败：错误提示信息 |

#### FR-AUTH-002 用户注册

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |
| 描述 | 管理员创建新的系统用户 |

**功能规则：**

1. 仅管理员角色可创建新用户
2. 用户角色分为：`admin`（管理员）、`auditor`（审计人员）、`viewer`（查看者）
3. 用户名不可重复
4. 密码长度不少于 6 位

#### FR-AUTH-003 修改密码

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |
| 描述 | 用户修改自己的登录密码 |

**功能规则：**

1. 需验证当前密码正确性
2. 新密码长度不少于 6 位
3. 密码修改后需重新登录

#### FR-AUTH-004 用户管理

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |
| 描述 | 管理员查看、编辑、删除用户 |

**功能规则：**

1. 支持用户列表查看
2. 支持编辑用户信息（姓名、角色、部门、邮箱、电话）
3. 支持启用/禁用用户账户
4. 不允许删除最后一个管理员账户

### 3.2 审计项目管理

#### FR-PROJ-001 创建项目

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |
| 描述 | 创建新的审计项目 |

**数据字段：**

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| name | TEXT | 是 | 项目名称 |
| type | ENUM | 是 | 项目类型：routine（常规审计）/ special（专项审计）/ follow-up（跟踪审计）/ other（其他） |
| status | ENUM | 是 | 项目状态：planning / executing / reviewing / completed / archived，默认 planning |
| description | TEXT | 否 | 项目描述 |
| audit_object | TEXT | 否 | 被审计单位 |
| audit_scope | TEXT | 否 | 审计范围和重点方向 |
| audit_period_start | TEXT | 否 | 审计期间开始日期 |
| audit_period_end | TEXT | 否 | 审计期间结束日期 |
| audit_team | TEXT | 否 | 审计团队 |
| leader_id | TEXT FK | 否 | 项目负责人（关联 users 表） |
| budget_hours | REAL | 否 | 预算工时 |
| risk_level | ENUM | 否 | 风险等级：low / medium / high / critical |

**功能规则：**

1. 项目创建时自动生成 UUID 作为主键
2. 自动生成项目编号（code 字段，唯一）

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |

**功能规则：**

1. 支持按项目类型、状态筛选

1. 各标签页采用懒加载策略，切换时才加载数据

3. 支持项目删除（级联删除关联的访谈、风险点、待办等）

#### FR-PROJ-004 项目统计

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |

### 3.5 风险点管理

#### FR-RISK-001 创建风险点

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |
| 描述 | 在项目下创建新的风险点 |

**数据字段：**

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| project_id | TEXT FK | 是 | 所属项目 |
| title | TEXT | 是 | 风险标题 |
| category | ENUM | 是 | 分类：financial / operational / compliance / strategic / information / other |
| risk_level | ENUM | 是 | 等级：low / medium / high / critical |
| likelihood | ENUM | 是 | 可能性：rare / unlikely / possible / likely / almost_certain |
| impact | ENUM | 是 | 影响：insignificant / minor / moderate / major / catastrophic |
| source | TEXT | 否 | 风险来源 |
| related_interview_id | TEXT FK | 否 | 关联访谈记录 |
| evidence | TEXT | 否 | 证据描述 |
| mitigation_plan | TEXT | 否 | 缓释方案 |
| responsible_person | TEXT | 否 | 责任人 |
| due_date | TEXT | 否 | 截止日期 |
| status | ENUM | 否 | 状态：identified / analyzing / mitigating / resolved / accepted / transferred |

**功能规则：**

1. 所有可能性、影响程度、分类、等级字段均通过下拉选择输入，值必须符合数据库 CHECK 约束
2. 自动记录创建时间
3. 支持关联到具体的访谈记录

#### FR-RISK-002 风险矩阵可视化

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |
| 描述 | 以 5×5 矩阵形式展示风险分布 |

**矩阵定义：**

|  | 罕见 (rare) | 不太可能 (unlikely) | 可能 (possible) | 很可能 (likely) | 几乎确定 (almost_certain) |
|--|------------|-------------------|----------------|---------------|------------------------|
| **灾难性** (catastrophic) | 中 | 高 | 高 | 严重 | 严重 |
| **重大** (major) | 低 | 中 | 高 | 高 | 严重 |
| **中等** (moderate) | 低 | 中 | 中 | 高 | 高 |
| **轻微** (minor) | 低 | 低 | 中 | 中 | 高 |
| **可忽略** (insignificant) | 低 | 低 | 低 | 中 | 中 |

**颜色编码：**

| 风险等级 | 颜色 | 阈值 |
|---------|------|------|
| 低 | 绿色 `#e8f5e9` | 得分 ≤ 3 |
| 中 | 黄色 `#fff9c4` | 得分 4-6 |
| 高 | 橙色 `#ffe0b2` | 得分 7-12 |
| 严重 | 红色 `#ffcdd2` | 得分 > 12 |

**功能规则：**

1. 横轴为可能性（从左到右递增）
2. 纵轴为影响程度（从上到下递减）
3. 每个单元格显示落入该区域的风险点数量
4. 点击单元格可查看对应风险点详情
5. 矩阵下方以表格形式列出所有风险点的详细信息

#### FR-RISK-003 风险状态管理

| 属性 | 说明 |
|------|------|
| 优先级 | 中 |
| 描述 | 跟踪风险点的处理状态 |

**状态定义：**

| 状态 | 说明 |
|------|------|
| identified | 已识别：新发现的风险点 |
| analyzing | 分析中：正在评估风险详情 |
| mitigating | 缓解中：正在执行缓释方案 |
| resolved | 已解决：风险已消除或降至可接受水平 |
| accepted | 已接受：管理层决定接受该风险 |
| transferred | 已转移：风险已转移给第三方 |

### 3.6 AI 智能分析

#### FR-AI-001 AI 服务配置

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |
| 描述 | 配置 AI 服务提供商和 API Key |

**支持的提供商：**

| 提供商 | 默认模型 | API 地址 |
|--------|---------|---------|
| DeepSeek | deepseek-chat | https://api.deepseek.com/v1 |
| 通义千问 | qwen-turbo | https://dashscope.aliyuncs.com/compatible-mode/v1 |
| 腾讯元宝 | hunyuan-pro | https://api.hunyuan.cloud.tencent.com/v1 |
| 豆包 | doubao-pro-32k | https://ark.cn-beijing.volces.com/api/v3 |

**功能规则：**

1. API Key 使用 AES-256-CBC 加密后存储在本地数据库
2. 支持测试连接功能，验证 API Key 有效性
3. 支持切换不同提供商和模型
4. 支持配置多个提供商的降级顺序

#### FR-AI-002 单条访谈 AI 辅助分析

| 属性 | 说明 |
|------|------|
| 优先级 | 高 |

| AI 服务 | `ai:` | 13 | AI 调用 + 配置管理 |
| 待办事项 | `todos:` | 5 | CRUD |
| 附件 | `attachments:` | 3 | 创建、列表、删除 |
| 设置 | `settings:` | 4 | 系统设置 CRUD |
| 日志 | `logs:` | 2 | 列表、清除 |
| 备份 | `backup:` | 4 | 创建、恢复、列表、删除 |

**接口总计：约 69 个**

### 5.2 统一返回格式

所有 IPC 接口返回统一的 JSON 格式：

```javascript
// 成功
{ success: true, data: <数据>, id: <新建记录ID> }

// 失败
{ success: false, message: <错误信息> }
```

### 5.3 CRUD 标准模式

所有数据模块遵循统一的 CRUD 模式：

```
<module>:create    → (data) → { success, id }
<module>:getById   → (id) → { success, data }
<module>:list      → (filters) → { success, data: [] }
<module>:update    → (id, updates) → { success }
<module>:delete    → (id) → { success }
```

**通用过滤参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| project_id | TEXT | 按项目筛选 |
| keyword | TEXT | 关键词模糊搜索 |
| status | TEXT | 按状态筛选 |
| _page | INTEGER | 分页页码 |
| _limit | INTEGER | 每页数量 |
